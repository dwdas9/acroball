# M3 - Merge Workflow

## Objective

Deliver the first complete user-facing PDF workflow: merge multiple PDFs into
one output file.

## Features Implemented

- Merge tool page.
- Add files through file picker.
- Add files through drag/drop.
- Reorder files through buttons.
- Reorder files through drag/drop.
- Output folder selection.
- Editable output file name.
- Output path preview.
- Validation for missing inputs, duplicate files, missing folder and invalid
  file names.
- Progress reporting.
- Cancellation.
- Success and failure result states.
- Open output folder action.
- Reset workflow for another merge.

## Architecture Summary

M3 added a shared job execution model:

- `IJobExecutor`
- `JobRunner`
- `JobRequestBase`
- `JobExecutionContext`
- `JobExecutionResult`
- `JobProgress`
- `MergeJobRequest`
- `MergeJob`

`MergeViewModel` depends on Application abstractions and `IFileDialogService`.
`MergeView` owns Avalonia drag/drop event details.

## Dependencies

No runtime dependency was added specifically for M3. UI tests use Moq, which is
already centrally managed.

## Manual Verification

- Run the app.
- Open Merge.
- Add two or more PDFs.
- Reorder files.
- Choose an output folder.
- Edit the suggested output file name.
- Run merge and confirm success state.
- Confirm the output PDF exists.
- Start another merge and confirm state resets.
- Try a missing file and confirm validation appears.

## Known Issues

- Merge UI does not expose per-file passwords.
- No rendered UI smoke tests are present.
- Some display/source text still contains encoding artifacts.
- The tool catalog labels Split, Rotate and Extract as M2, but those pages are
  still placeholders.

## Lessons Learned

- A shared job runner gives a good pattern for future long-running workflows.
- ViewModel tests cover most merge behavior without requiring Avalonia
  rendering.
- Future workflows should reuse validation, progress and cancellation patterns
  from Merge.
